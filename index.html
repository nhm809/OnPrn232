Những gì cần vững để làm tốt bài thi PRN232

Đề thi có dạng 2 project độc lập — Q1 xây Web API (EF Core + SQL Server), Q2 xây MVC/Razor gọi sang Q1 qua HTTP. Tài liệu này gom lại kiến thức nền tảng cần vững, kỹ năng thao tác tay, và toàn bộ lỗi thật đã gặp trong lúc luyện tập.

00Tổng quan đề thi

Nắm được hình dạng chung của đề trước khi học sâu từng phần.

Q1 — Web API Bắt buộc

4 endpoint tự viết theo yêu cầu đề (GET danh sách, GET lọc nâng cao, POST tạo mới, DELETE), cộng thêm một số "Provided APIs" — API có sẵn mô tả trong đề, bạn phải tự cài đặt luôn vì solution không có sẵn code. Dữ liệu lấy từ SQL Server qua EF Core.

Q2 — MVC/Razor Bắt buộc

1–2 trang Razor, KHÔNG được đọc DB trực tiếp — chỉ được gọi sang Q1 qua HttpClient. Có thể phải dùng 1 class tiện ích có sẵn (kiểu Utilities.GetAbsoluteUrl) để ghép URL, không được tự ghép chuỗi tuỳ ý.

✓

Điểm mấu chốt: Q1 và Q2 là 2 process độc lập, chạy song song lúc test. Q1 phải chạy đúng trước và đứng vững một mình (test bằng Swagger) rồi mới ráp Q2 vào — đừng code cả hai cùng lúc, sẽ không biết lỗi nằm ở đâu.

Nền tảng để viết đúng 4 endpoint của Q1.

Controller tối giản, đúng route

// Route đầy đủ trên từng action nếu URL không cùng 1 tiền tố
[ApiController]
[Route("api")]

02EF Core & quan hệ dữ liệu Bắt buộc

Phần hay gây rối nhất: phân biệt N-N thuần và N-N có dữ liệu phụ.

Database First: Scaffold-DbContext

Đề cho sẵn DB → dùng dotnet ef dbcontext scaffold để tự sinh Model + DbContext, không viết tay. Lệnh chuẩn (xem chi tiết ở mục 06):

dotnet ef dbcontext scaffold "Server=.\SQLEXPRESS;Database=TenDB;User Id=...;Password=...;TrustServerCertificate=True;" Microsoft.EntityFrameworkCore.SqlServer -o Models --context-dir Data -c AppDbContext -d --no-onconfiguring -f

Hai kiểu quan hệ N-N — khác nhau hoàn toàn khi viết LINQ

N-N thuần Nên biết

Bảng trung gian chỉ có 2 khóa ngoại, không cột phụ (vd RoomTypeServices). EF Core Scaffold tự nuốt mất bảng này, sinh ra ICollection 2 chiều trực tiếp. Include chỉ cần 1 cấp: .Include(rt => rt.Services).

N-N có dữ liệu phụ Nên biết

Bảng trung gian có thêm cột (vd NightCount trong BookingDetails). EF Core giữ nguyên thành entity riêng, khóa phức hợp. Include phải xuyên 2 cấp bằng .ThenInclude().

Include = JOIN thật, nhưng có 2 điều cần nhớ

!

Không gọi Include → navigation property không lỗi, chỉ rỗng. Code chạy được, không exception, nhưng field sai (0 hoặc rỗng) — lỗi rất khó phát hiện nếu không để ý.

Include không dùng để lọc. Muốn lọc theo quan hệ N-N, dùng .Where(x => x.Services.Any(s => s.ServiceId == id)), không phải Include.

Composite key & Unique index (nếu tự viết OnModelCreating)

modelBuilder.Entity<BookingDetail>()
    .HasKey(bd => new { bd.BookingId, bd.RoomId });

modelBuilder.Entity<Room>()
    .HasIndex(r => r.RoomNumber)
    .IsUnique();

03LINQ cheat sheet Bắt buộc

7 pattern này phủ gần hết mọi API kiểu "danh sách + lọc + tính toán".

Lọc động theo tham số tuỳ chọn

var query = _context.Rooms.AsQueryable();
if (!string.IsNullOrWhiteSpace(status))
    query = query.Where(r => r.Status.ToLower() == status.ToLower());
if (minPrice.HasValue)
    query = query.Where(r => r.BasePrice >= minPrice.Value);
var result = await query.ToListAsync();

Đếm / Tổng trên navigation property

ServiceCount = rt.Services.Count                          // đếm N-N
TotalNights  = r.BookingDetails.Sum(bd => bd.NightCount)  // tổng 1-N

Lấy bản ghi "mới nhất" theo 1 mốc thời gian

CurrentGuest = r.BookingDetails
    .OrderByDescending(bd => bd.Booking!.CheckInDate)
    .Select(bd => bd.Booking!.Guest!.FullName)
    .FirstOrDefault() ?? "No Guest";

Kiểm tra tồn tại / trùng lặp

bool exists = await _context.Rooms.AnyAsync(r => r.RoomNumber == dto.RoomNumber);

Group + tính tổng theo nhóm

var grouped = await _context.BookingDetails
    .GroupBy(bd => bd.RoomId)
    .Select(g => new { RoomId = g.Key, TotalNights = g.Sum(x => x.NightCount) })
    .ToListAsync();

Chiếu (Select) thẳng sang DTO — nên dùng khi có thể

var dto = await _context.RoomTypes
    .Select(rt => new RoomTypeSummaryDto
    {
        RoomTypeId = rt.RoomTypeId,
        ServiceCount = rt.Services.Count
    })
    .ToListAsync();

Dịch được thẳng sang SQL, không load dư dữ liệu. Nhưng logic phức tạp nhiều tầng (như CurrentGuest ở trên) thường phải ToListAsync() trước rồi mới .Select() lần 2 trên memory (LINQ to Objects) — EF Core khó dịch biểu thức lồng quá sâu sang SQL.

Phân trang

var paged = await query.Skip((page - 1) * pageSize).Take(pageSize).ToListAsync();

04MVC / Razor Bắt buộc

Q2 chỉ là lớp hiển thị — không đọc DB trực tiếp, chỉ gọi HTTP sang Q1 để lấy dữ liệu rồi render HTML.

MVC là gì? — 3 thành phần

Model (ViewModel)

Class C# định nghĩa cấu trúc dữ liệu. Trong Q2, đây là ViewModel — class chứa dữ liệu nhận từ API, đặt trong Q2/Models/

View (.cshtml)

File HTML + Razor syntax. Chỉ hiển thị, không chứa logic xử lý. Đặt trong Q2/Views/{Controller}/

Controller

Điều phối: nhận request từ user → gọi API (Q1) → lấy dữ liệu → đưa vào View để render HTML trả về

Luồng request trong bài thi

  1. User vào trang /RoomType

    Trình duyệt gửi GET request đến Q2 (MVC).

  2. RoomTypeController.Index() được gọi

    ASP.NET Core dựa vào URL để tìm đúng Controller + Action.

  3. Controller gọi API sang Q1 bằng HttpClient

    Utilities.GetAbsoluteUrl("/api/roomtypes") → GetFromJsonAsync<List<...>>(url)

  4. Q1 trả về JSON, Controller parse thành object C#

    GetFromJsonAsync tự động deserialize JSON → List<RoomTypeVm>

  5. Controller gọi return View(list)

    Truyền data vào View — file .cshtml cùng tên action sẽ được render.

  6. View dùng Razor để render HTML

    @foreach, @if, @Model.Property — HTML cuối cùng trả về trình duyệt.

✓

Q2 không truy cập database trực tiếp. Mọi dữ liệu đều gọi qua API của Q1 bằng HttpClient. Q1 và Q2 là 2 process chạy song song.

Cú pháp Razor cơ bản — ký tự @

Hiển thị giá trị: @biểuthức — giá trị C# tự động HTML-encode (an toàn XSS).

<p>Xin chào, @Model.FullName</p>
<p>Hôm nay là @DateTime.Now.ToString("dd/MM/yyyy")</p>

Khối lệnh C#: @{ ... } — viết nhiều dòng C#, khai báo biến.

@{
    var message = "Hello";
    var count = Model.Count;
    ViewData["Title"] = "Danh sách";
}

Điều kiện: @if / else

@if (Model.Count > 0)
{
    <p>Có @Model.Count kết quả</p>
}
else
{
    <p>Không có dữ liệu</p>
}

Vòng lặp: @foreach — dùng nhiều nhất trong thi, hiển thị bảng dữ liệu.

<table>
  <tr><th>Tên</th><th>Giá</th></tr>
  @foreach (var item in Model)
  {
      <tr>
          <td>@item.Name</td>
          <td>@item.Price.ToString("N0")</td>
      </tr>
  }
</table>

Khai báo Model cho View: @model — luôn ở dòng đầu tiên của file .cshtml.

@model List<RoomTypeVm>

// Sau khi khai báo, "Model" = biến có kiểu List<RoomTypeVm>
<p>Tổng: @Model.Count loại phòng</p>
✕

@model (chữ thường) ≠ @Model (chữ hoa). @model = khai báo kiểu (1 lần, dòng đầu). @Model = truy cập dữ liệu (dùng xuyên suốt file). Nhầm 2 cái này là lỗi build!

Controller — cấu trúc và routing

using System.Net.Http.Json;
using Q2.Models;
using Microsoft.AspNetCore.Mvc;

namespace Q2.Controllers;

public class RoomTypeController : Controller
{
    private readonly HttpClient _http;

    public RoomTypeController(HttpClient http) => _http = http;

    // GET: /RoomType hoặc /RoomType/Index
    public async Task<IActionResult> Index()
    {
        var url = Utilities.GetAbsoluteUrl("/api/roomtypes");
        var list = await _http.GetFromJsonAsync<List<RoomTypeVm>>(url) ?? new();
        return View(list);
    }
}

Q2 gọi API của Q1 — HttpClient

Trong bài thi, Q2 gọi sang Q1 bằng HttpClient. URL được tạo bằng Utilities.GetAbsoluteUrl() — helper của trường.

GET — Lấy danh sách:

var url = Utilities.GetAbsoluteUrl("/api/roomtypes");
var list = await _http.GetFromJsonAsync<List<RoomTypeVm>>(url);
return View(list ?? new List<RoomTypeVm>());

GET — Với query string (tìm kiếm/lọc):

var url = Utilities.GetAbsoluteUrl(
    $"/api/filter-rooms?status={status}&minPrice={minPrice}");
var rooms = await _http.GetFromJsonAsync<List<RoomVm>>(url);

POST — Gửi dữ liệu tạo mới:

var url = Utilities.GetAbsoluteUrl("/api/rooms");
var response = await _http.PostAsJsonAsync(url, newRoom);
if (response.IsSuccessStatusCode)
    return RedirectToAction("Index");

// Nếu lỗi, đọc thông báo lỗi từ API
var error = await response.Content.ReadAsStringAsync();
ViewBag.Error = error;
return View(model);

DELETE — Xoá:

var url = Utilities.GetAbsoluteUrl($"/api/rooms/{id}");
var response = await _http.DeleteAsync(url);
return RedirectToAction("Index");

Controller — Nhận tham số từ form:

public async Task<IActionResult> Index(
    string? status, decimal? minPrice)
{
    // Xây query string gửi sang Q1
    var qs = "";
    if (!string.IsNullOrEmpty(status))
        qs += $"&status={status}";
    if (minPrice.HasValue)
        qs += $"&minPrice={minPrice}";
    if (qs.Length > 0) qs = "?" + qs.Substring(1);

    var url = Utilities.GetAbsoluteUrl($"/api/filter-rooms{qs}");
    var list = await _http.GetFromJsonAsync<List<RoomVm>>(url)
              ?? new();

    // Giữ lại giá trị đã chọn trên form
    ViewBag.Status = status;
    ViewBag.MinPrice = minPrice;

    return View(list);
}
✓

Form dùng method="get" → tham số đi vào URL query string → Controller nhận qua parameter name khớp. Không cần [HttpPost]. Dùng ViewBag để giữ giá trị đã chọn sau khi search.

06Kỹ năng thao tác tay Kỹ năng tay

Đây là phần khác biệt lớn nhất giữa "biết lý thuyết" và "làm được trong phòng thi" — luyện tay cho quen, đừng để mất thời gian thi vì lỗi vặt.

Xử lý DB không xoá được

USE master;
GO
IF DB_ID('TenDB') IS NOT NULL
BEGIN
    ALTER DATABASE TenDB SET SINGLE_USER WITH ROLLBACK IMMEDIATE;
    DROP DATABASE TenDB;
END
GO

Nguyên nhân hay gặp: 1 tab Query khác trong SSMS (hoặc chính cửa sổ đang chạy script) vẫn đang "đứng trong" DB đó. USE master; ở đầu script + SET SINGLE_USER WITH ROLLBACK IMMEDIATE giải quyết 95% trường hợp.

Chạy script nhanh bằng sqlcmd

sqlcmd -S ".\SQLEXPRESS" -U user -P pass -i "C:\duong\dan\script.sql"

Dùng dấu \ kiểu Windows cho đường dẫn file — dùng / sẽ bị lỗi "Access denied".

PMC vs CLI khi Scaffold — không được trộn cú pháp

Package Manager Console

Scaffold-DbContext "..." Microsoft.EntityFrameworkCore.SqlServer -OutputDir Models -Context AppDbContext -DataAnnotations -NoOnConfiguring -Force

Terminal (dotnet ef)

dotnet ef dbcontext scaffold "..." Microsoft.EntityFrameworkCore.SqlServer -o Models -c AppDbContext -d --no-onconfiguring -f

PMC dùng PascalCase 1 gạch ngang; CLI dùng kebab-case 2 gạch ngang. Trộn 2 kiểu → lỗi Unrecognized option.

Verify bằng file publish, không tin F5

dotnet publish -c Release -o ./Q1_MaSV
cd Q1_MaSV
dotnet Q1.dll --urls http://localhost:5100
# mở /swagger — phải load được, đúng như lúc thầy sẽ kiểm tra

07Bẫy hay gặp Từ buổi luyện tập thật

Toàn bộ mục này là lỗi thật đã xảy ra và đã kiểm chứng cách sửa — không phải lý thuyết suông.

✕

Swagger biến mất sau khi publish. Nguyên nhân: UseSwagger() bị bọc trong if (app.Environment.IsDevelopment()). Chạy file .dll publish trực tiếp mặc định là Production, không phải Development.

Sửa: gọi UseSwagger() / UseSwaggerUI() không điều kiện.

✕

CreatedAtAction(nameof(CreateRoom), ...) tự trỏ vào chính nó. Vì không có action GET-theo-id nào để trỏ tới, header Location sinh ra sai/vô nghĩa.

Sửa: dùng return StatusCode(201, entity); đơn giản.

✕

appsettings.Development.json không được nạp khi chạy bản publish (vì ASPNETCORE_ENVIRONMENT mặc định là Production).

Sửa: connection string thật phải nằm trong appsettings.json gốc.

!

dotnet add package ... -Version x bị bỏ qua âm thầm — flag đúng là -v hoặc --version, viết hoa V khiến CLI không báo lỗi rõ ràng, chỉ in trang help và không set version.

!

Quên --no-onconfiguring khi scaffold → EF nhúng cứng connection string thật (kể cả password) vào code sinh ra trong OnConfiguring() — vi phạm thẳng yêu cầu "connection string phải chỉnh được qua appsettings.json".

!

Namespace rỗng làm lỗi build. @using Q2.Models trong _ViewImports.cshtml sẽ lỗi nếu Models/ chưa có file .cs nào — namespace C# chỉ "tồn tại" khi có ít nhất 1 class bên trong.

!

dotnet ef cần project build được trước mới chạy scaffold được. Nếu Program.cs đã lỡ tham chiếu 1 DbContext chưa tồn tại, phải comment lại phần đó, scaffold xong mới mở lại.

08Quy trình chuẩn lúc thi

  1. Chạy script SQL đề cho

    Thêm USE master; + SET SINGLE_USER trước DROP DATABASE nếu script gốc chưa có.

  2. Sửa Q1/appsettings.json

    Key JSON phải đúng tên đề yêu cầu (thường là ConnectionStrings:MyCnn). Chỉ đổi phần Database=.

  3. Scaffold DB (nếu làm Database First)

    Nhớ --no-onconfiguring, đúng flag PMC hoặc CLI, không trộn.

  4. Viết Controller/Service cho Q1, test bằng Swagger

    Test xong hết 4 endpoint + provided APIs trước khi đụng vào Q2.

  5. Chạy Q1 đúng port mà Q2 đang trỏ tới

    Xem Q2/appsettings.json mục GivenAPIBaseUrl — Q1 phải chạy đúng port đó (không phải port ngẫu nhiên trong launchSettings).

  6. Viết Controller/View cho Q2, test song song với Q1 đang chạy

    Cả 2 project chạy cùng lúc (2 terminal riêng, hoặc Multiple Startup Projects trong Visual Studio).

  7. Publish cả 2, verify lại bằng file .dll

    dotnet publish -c Release -o ./Q1_MaSV rồi chạy trực tiếp — đừng chỉ tin F5.

09Checklist ôn tập trước khi thi

  • Viết được Controller API trả JSON đúng status code (200/201/204/400/404) không cần tra cứu
  • Phân biệt được N-N thuần và N-N có dữ liệu phụ chỉ bằng cách nhìn bảng trung gian
  • Viết được Include/ThenInclude đúng cấu trúc không cần thử sai
  • Nhớ 7 pattern LINQ ở mục 03 mà không cần mở tài liệu
  • Tự sửa được lỗi "Cannot drop database ... currently in use" trong dưới 1 phút
  • Phân biệt được flag PMC và CLI khi scaffold, không lẫn lộn
  • Biết chạy Q1/Q2 song song đúng port, không nhầm
  • Luôn test lại bằng file publish trước khi nộp, không tin F5
  • Viết được Razor view với id HTML khớp chính xác pattern đề cho
  • Biết dùng Utilities.GetAbsoluteUrl (hoặc tương đương) thay vì tự ghép URL
Tổng hợp từ quá trình luyện tập thực tế — mọi lệnh, lỗi và cách sửa trong tài liệu này đều đã được chạy thử và kiểm chứng, không phải suy đoán lý thuyết.
